# LogAnalysis

로그 분석 프로그램 팀 프로젝트 저장소입니다.

이 문서는 **팀원들이 Git을 처음 사용하는 상황**을 기준으로, 저장소를 어떻게 사용하면 되는지 한 번에 볼 수 있도록 정리한 협업 가이드입니다.

---

## 1. 프로젝트 목적

로그 파일을 읽어 다음과 같은 정보를 분석하는 프로그램을 구현합니다.

- 최다 사용 key 분석
- 브라우저별 접속 횟수 / 비율 분석
- 응답 코드(200, 404, 403, 500) 분석
- 요청이 가장 많은 시간 분석
- 범위 라인(예: 1000~1500) 기준 분석
- 결과 화면 출력
- 결과 파일(report) 생성

---

## 2. Git / GitHub / GUI 도구란?

### Git
프로젝트 파일의 **변경 이력**을 관리하는 도구입니다.

### GitHub
Git으로 관리한 프로젝트를 **온라인에 올려두는 저장소**입니다.

### GUI 도구
Git 명령어를 직접 입력하지 않고도, 버튼 중심으로 Git을 사용할 수 있게 도와주는 도구입니다.

예:
- GitFork
- SourceTree
- GitHub Desktop
- IntelliJ / Eclipse의 Git 기능

> 우리 팀은 **GitHub를 기준으로 협업**하며, Git을 다룰 때 사용하는 **GUI 도구는 자유롭게 선택**할 수 있습니다.

---

## 3. 왜 Git을 써야 하나?

Git을 사용하면 다음이 가능합니다.

- 누가 어떤 파일을 수정했는지 확인 가능
- 이전 상태로 되돌리기 가능
- 각자 맡은 기능을 따로 작업 가능
- 팀원 작업을 하나로 합치기 가능

Git을 사용하지 않으면 보통 다음 문제가 생깁니다.

- `최종`, `진짜최종`, `최종수정본` 같은 파일이 계속 생김
- 누가 최신 버전인지 알기 어려움
- 같은 파일을 여러 명이 수정해서 충돌 발생

---

## 4. 꼭 알아야 하는 핵심 개념

### Repository
프로젝트와 변경 기록이 저장되는 공간입니다.

- **Remote Repository**: GitHub 저장소
- **Local Repository**: 내 컴퓨터에 복사된 저장소

### Commit
작업 내용을 **기록으로 저장**하는 것입니다.

예시:
- 로그인 기능 추가
- 로그 파싱 기능 구현
- 출력 형식 수정

### Push
내 컴퓨터에서 commit한 내용을 GitHub에 올리는 것입니다.

### Pull
GitHub의 최신 변경사항을 내 컴퓨터로 가져오는 것입니다.

### Branch
기능별로 나누어 작업하는 독립 공간입니다.

### Merge
다른 branch에서 작업한 내용을 main에 합치는 것입니다.

### Conflict
같은 파일의 같은 부분을 여러 명이 수정해서 Git이 자동으로 합치지 못하는 상태입니다.

## 5. 우리 팀 협업 방식

### 기본 원칙
- `main` 브랜치는 **최종본만 유지**합니다.
- 각자 **자기 브랜치에서만 작업**합니다.
- 작업 시작 전 반드시 **Pull** 합니다.
- 작업이 끝나면 **Commit → Push** 합니다.
- 최종 병합은 **메인 담당 또는 팀장**이 진행합니다.

---

## 6. 브랜치 규칙

팀 역할에 맞춰 브랜치를 다음과 같이 사용합니다.

- `main` : 최종 통합본
- `feature/main-flow` : 메인 흐름 담당
- `feature/auth` : 인증 담당
- `feature/menu` : 메뉴 담당
- `feature/parser` : 파싱 담당
- `feature/output-export` : 출력 및 내보내기 담당

### 브랜치 생성 규칙
- 형식: `feature/기능이름`
- 예시:
  - `feature/auth`
  - `feature/parser`
  - `feature/output-export`

---

## 7. 팀원별 담당 범위

### 메인
- 프로그램 전체 실행 흐름 제어
- 각 기능 연결
- 최종 통합 테스트

### 인증
- 로그인 처리
- 계정 및 권한 처리

### 메뉴
- 사용자 입력 처리
- 메뉴 선택 및 파일 선택(`JFileChooser`) 처리

### 파싱
- 로그 파일 읽기
- 로그 데이터를 분석 가능한 형태로 변환

### 출력 / 내보내기
- 결과 화면 출력
- report 파일 생성 및 저장

## 8. 작업 순서

### 처음 1회
1. 저장소 Clone
2. 사용하는 Git GUI 도구에서 저장소 열기
3. 자신의 브랜치 생성
4. 자신의 브랜치로 Checkout

### 매일 작업 시작 전
1. 사용하는 Git GUI 도구 실행
2. 현재 브랜치 확인
3. `Pull` 해서 최신 코드 받기

### 작업 중
1. 맡은 기능 구현
2. 변경 파일 확인
3. 필요한 파일만 Stage

### 작업 종료 시
1. Commit 작성
2. Push
3. 팀장 또는 메인 담당에게 공유

### 한 줄 요약
**Pull → 작업 → Stage → Commit → Push**

## 9. Eclipse에서 프로젝트 가져오는 방법

레포에 올라간 **공통 프로젝트를 기준으로 작업**해야 하므로, 각자 새 프로젝트를 따로 만들지 말고 아래 순서로 가져옵니다.

### 방법 1. Existing Projects into Workspace
가장 먼저 이 방법을 시도합니다.

1. 저장소를 로컬에 clone 합니다.
2. Eclipse를 실행합니다.
3. `File > Import...` 로 들어갑니다.
4. `General > Existing Projects into Workspace` 를 선택합니다.
5. `Select root directory` 에 clone 받은 폴더를 지정합니다.
6. 프로젝트 목록에 `LogAnalysis`가 보이면 체크 후 `Finish` 합니다.

### 방법 2. Import가 안 될 때
프로젝트가 목록에 보이지 않으면 아래 방법으로 진행합니다.

1. `File > New > Java Project` 로 들어갑니다.
2. `Project name` 을 `LogAnalysis`로 입력합니다.
3. `Use default location` 체크를 해제합니다.
4. clone 받은 저장소 폴더를 위치로 지정합니다.
5. `Finish` 합니다.

### 가져온 뒤 확인할 것
- `src` 폴더가 정상적으로 보이는지 확인
- 빨간 X 에러가 없는지 확인
- 프로젝트 기준은 항상 **레포에 올라간 공통 프로젝트**인지 확인

> 주의: 각자 개인 프로젝트를 따로 만들어 작업하지 말고, 반드시 **레포 기준 프로젝트** 안에서만 작업합니다.

## 10. GUI 도구에서 자주 쓰는 기능

- **Clone**: 저장소를 내 컴퓨터로 복사
- **Checkout**: 브랜치 이동
- **Pull**: 최신 코드 가져오기
- **Stage**: commit에 포함할 파일 선택
- **Commit**: 변경사항 저장
- **Push**: GitHub에 업로드
- **Merge**: 브랜치 합치기

---

## 11. Commit 메시지 규칙

Commit 메시지는 **무엇을 했는지 한 줄로 알 수 있게** 작성합니다.

### 형식
`타입: 작업 내용`

### 예시
- `feat: 로그인 인증 기능 추가`
- `feat: 로그 파서 구현`
- `feat: 결과 파일 저장 기능 추가`
- `fix: 브라우저 비율 계산 오류 수정`
- `refactor: MainApp 실행 흐름 정리`
- `docs: README 협업 규칙 추가`

### 자주 쓰는 타입
- `feat` : 기능 추가
- `fix` : 오류 수정
- `refactor` : 구조 개선
- `docs` : 문서 수정
- `test` : 테스트 코드 추가

---

## 12. 공통으로 지킬 규칙

### 필수 규칙
1. `main` 브랜치에서 직접 작업하지 않는다.
2. 작업 시작 전에 반드시 `Pull` 한다.
3. 자기 담당 브랜치에서만 작업한다.
4. commit은 **기능 단위**로 한다.
5. commit 메시지는 의미 있게 작성한다.
6. 다른 사람이 수정한 파일을 함부로 건드리지 않는다.
7. 병합 전에는 반드시 실행 확인을 한다.
8. 충돌이 나면 혼자 무리해서 해결하지 말고 팀장과 함께 확인한다.

### 금지 사항
- 의미 없는 commit 메시지 작성
  - 예: `수정`, `완성`, `123`
- 여러 기능을 한 commit에 몰아넣기
- `main`에 바로 push 하기
- 실행 확인 없이 merge 요청하기

## 13. 충돌(conflict)이 났을 때

충돌은 같은 파일의 같은 부분을 둘 이상 수정했을 때 발생합니다.

### 해결 순서
1. 어떤 파일에서 충돌이 났는지 확인
2. 누가 어떤 부분을 수정했는지 확인
3. 필요한 내용만 남기고 코드 정리
4. 다시 commit
5. 팀원과 함께 최종 확인

### 주의
충돌이 났을 때는 바로 덮어쓰지 말고, **어느 코드가 최종으로 남아야 하는지 먼저 확인**합니다.

---

## 14. 업로드하지 말아야 할 것

다음 파일은 가급적 저장소에 올리지 않습니다.

- 실행 결과물
- 개인 설정 파일
- IDE 개인 환경 파일
- 불필요한 캐시 파일

예:
- `.class`
- `bin/`
- `.idea/`
- `.settings/`
- 개인 PC 경로가 들어간 설정 파일

> 추후 `.gitignore` 파일도 함께 설정하는 것을 권장합니다.

## 15. 병합 방식

- 팀원은 자신의 기능 브랜치에서 작업합니다.
- 작업이 완료되면 Push 후 팀장/메인 담당에게 공유합니다.
- 최종 확인 후 `main` 브랜치에 병합합니다.

### 권장 방식
- 기능 구현 완료
- 실행 확인 완료
- 팀장 확인
- main 병합

---

## 16. 팀원 체크리스트

### 작업 시작 전
- [ ] 현재 브랜치를 확인했다.
- [ ] 최신 코드를 Pull 했다.

### 작업 종료 전
- [ ] 실행 확인을 했다.
- [ ] 불필요한 파일이 포함되지 않았는지 확인했다.
- [ ] Commit 메시지를 올바르게 작성했다.
- [ ] Push 완료했다.

## 17. 한눈에 보는 협업 흐름

```text
저장소 Clone
→ 내 브랜치 Checkout
→ Pull
→ 기능 구현
→ Stage
→ Commit
→ Push
→ 팀장 확인
→ main 병합
```

## 18. 참고

이 저장소는 팀 협업을 위한 공간이며, 코드 작성뿐 아니라 **역할 분담, 버전 관리, 통합 과정**까지 프로젝트의 일부로 간주합니다.

팀원 모두가 같은 규칙을 지켜야 충돌을 줄이고 안정적으로 프로젝트를 진행할 수 있습니다.
